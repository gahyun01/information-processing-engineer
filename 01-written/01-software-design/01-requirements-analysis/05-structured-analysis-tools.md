# 💻 05. 구조적 분석 도구

## 🔍 구조적 분석 기법 ( Structured Analysis )

### 📖 정의

구조적 분석 기법은 **자료의 흐름과 처리를 중심으로 사용자의 요구사항을 분석하는 방법**이다.

시스템을 기능 중심으로 분해하고, 도형 중심의 분석 도구를 활용하여 요구사항을 체계적으로 표현한다.

#### ✨ 특징

- 자료의 흐름과 처리를 중심으로 분석한다.
- 도형 중심의 분석 도구와 분석 절차를 이용하여 요구사항을 파악하고 문서화한다.
- 하향식(Top-Down) 방법을 사용하여 시스템을 세분화한다.
- 분석 과정에서 중복을 제거할 수 있다.
- 시스템 분석의 품질을 향상시키고 명세서 작성에 활용된다.

#### 📚 주요 분석 도구

- DFD (Data Flow Diagram)
- 자료사전 (Data Dictionary)
- CASE
- HIPO

## 🔄 자료 흐름도 ( DFD, Data Flow Diagram )

### 📖 정의

자료 흐름도(DFD)는 **시스템 내에서 자료가 이동하고 처리되는 과정을 도형 중심으로 표현하는 구조적 분석 도구**이다.

시스템의 프로세스와 자료 저장소 사이의 자료 흐름을 표현하며, 자료의 입력·처리·출력을 중심으로 시스템을 모델링한다.

#### ✨ 특징

- 구조적 분석 기법에서 사용된다.
- 데이터의 흐름과 처리 과정을 중심으로 표현한다.
- 시스템의 기능을 프로세스 단위로 표현한다.
- 제어의 흐름은 중요하게 표현하지 않는다.
- 시간 흐름을 명확하게 표현하기 어렵다.
- 자료 흐름 그래프(Data Flow Graph), 버블 차트(Bubble Chart)라고도 한다.

<br/>

### 🧩 DFD 구성 요소

> 📝 암기 : **프플스터**

| 구성 요소 | 기호 | 설명 |
|---|---|---|
| Process (처리기) | 원 | 입력된 데이터를 원하는 형태로 변환하여 출력 |
| Data Flow (자료 흐름) | 화살표 | 구성 요소 간 데이터 이동 표현 |
| Data Store (자료 저장소) | 평행선 | 데이터가 저장된 장소 표현 |
| Terminator (단말) | 사각형 | 시스템과 교신하는 외부 개체 |

<br/>

#### 📌 Process (처리기)

입력된 데이터를 원하는 형태로 변환하여 출력하는 처리 과정이다.

- 처리, 기능, 변환, 버블이라고도 한다.
- 원 안에 프로세스 이름을 작성한다.

<br/>

#### 📌 Data Flow (자료 흐름)

DFD 구성 요소 간에 이동하는 데이터를 나타낸다.

- 화살표 방향으로 데이터 흐름을 표현한다.

<br/>

#### 📌 Data Store (자료 저장소)

데이터가 저장되는 장소를 나타낸다.

<br/>

#### 📌 Terminator (단말)

프로세스 처리 과정에서 데이터가 발생하는 시작과 종료를 나타내는 외부 개체이다.

- 시스템과 데이터를 주고받는 외부 엔티티이다.
- 사각형 안에 외부 개체명을 작성한다.

<br/>

### ✏️ 작성 지침

- 자료 흐름은 처리를 거쳐 변환될 때마다 새로운 이름을 부여한다.
- 어떤 처리가 출력 자료를 생성하기 위해서는 입력 자료가 존재해야 한다.
- 상위 단계와 하위 자료 흐름도의 자료 흐름은 서로 일치해야 한다.
- 입력 자료가 존재한다고 반드시 출력 자료가 존재해야 하는 것은 아니다.

## 📚 자료사전 ( DD, Data Dictionary )

### 📖 정의

자료사전은 **자료 흐름도에 표현된 자료를 자세히 정의하고 기록한 문서**이다.

데이터에 대한 데이터인 **메타데이터(Metadata)**를 관리하며, 자료 요소·자료 흐름·자료 저장소의 의미와 관계를 명확하게 정의한다.

#### ✨ 특징

- 자료 흐름도에 표현된 자료를 체계적으로 관리한다.
- 자료 요소, 자료 흐름, 자료 저장소의 의미를 정의한다.
- 개발자와 사용자가 데이터 의미를 쉽게 이해하도록 지원한다.
- 자료 흐름도에 표현된 모든 자료는 자료사전에 정의되어야 한다.

<br/>

### 🎯 작성 목적

- 자료 용어의 의미를 명확하게 정의한다.
- 개발 참여자 간 데이터 의미를 공유한다.
- 자료 정의의 중복을 제거한다.
- 데이터 구조를 체계적으로 관리한다.

<br/>

### 🔣 자료사전 기호

| 기호 | 의미 |
|---|---|
| = | 정의 |
| + | 연결 |
| () | 생략 가능 |
| {} | 반복 |
| [] | 선택 |
| ** | 주석 |

<br/>

### ✨ 작성 원칙

#### 📌 자료 의미 기술

- 자료의 의미를 주석으로 명확하게 표현한다.
- 대상 시스템에서 사용하는 의미를 기준으로 작성한다.

#### 📌 자료 구성 항목 기술

- 구성 항목을 그룹으로 묶는다.
- 각 그룹에 의미 있는 이름을 부여한다.

#### 📌 동의어 규정

- 동일한 자료에 여러 이름이 사용되지 않도록 관리한다.

#### 📌 자료 정의 중복 제거

- 동일한 자료가 여러 이름으로 정의되지 않도록 관리한다.


## 🧩 모델링 기법 ( Modeling )

### 📖 모델 ( Model )

모델은 **객체, 시스템 또는 개념의 구조와 동작을 표현하기 위한 추상화된 표현 방식**이다.

복잡한 현실 세계를 단순화하여 기호, 그림, 다이어그램 등으로 표현하며 시스템에 대한 이해와 의사소통을 돕는다.

#### ✨ 특징

- 개발 대상 시스템을 추상화하여 표현한다.
- 이해관계자 간의 의사소통을 향상시킨다.
- 문제 상황에 대한 이해와 해결 방법을 설명하는 데 활용된다.
- 개발될 시스템에 대한 예측과 분석이 가능하다.
- 개념 모델은 문제 영역의 객체와 관계를 표현한다.


<br/>

### 📖 모델링 ( Modeling )

모델링은 **현실 세계의 문제를 특정 목적에 맞게 단순화하여 모델로 표현하는 과정**이다.

#### ✨ 특징

- 개발팀이 시스템에 대한 공통된 개념을 공유할 수 있도록 한다.
- 응용 문제를 이해하는 데 도움을 준다.
- 모델링 결과는 이후 설계 및 개발 과정에 영향을 준다.
- 구조적 방법론에서는 DFD, 자료사전 등을 사용한다.
- 객체 지향 방법론에서는 UML을 사용한다.
- 소프트웨어 요구사항 분석에서 핵심적인 과정이다.

## 🤖 분석 자동화 도구 ( CASE, Computer Aided Software Engineering )

### 📖 정의

CASE는 **소프트웨어 개발 과정에서 요구사항 분석, 설계, 구현, 테스트 등을 자동화하여 지원하는 도구**이다.

요구사항을 자동으로 분석하고 명세서를 작성할 수 있도록 지원하는 요구사항 분석 자동화 도구이다.

#### ✨ 등장 배경

- 소프트웨어 규모 증가에 따른 개발 복잡성 증가
- 소프트웨어 위기 해결 필요
- 시스템 재사용성과 유지보수성 향상 필요
- 사용자 요구사항과 실제 시스템 간의 차이 해결 필요


<br/>

### 🎯 CASE 사용 효과

- 표준화된 개발 환경을 제공한다.
- 문서화 품질을 향상시킨다.
- 요구사항 변경에 대한 추적이 쉽다.
- 소프트웨어 유지보수 비용을 감소시킨다.
- 개발자 간 협업과 의사소통을 향상시킨다.


<br/>

### 📊 CASE 분류

#### ⬆️ 상위 CASE

소프트웨어 개발 초기 단계에서 사용하는 도구이다.

#### 지원 기능

- 계획 수립
- 요구사항 분석
- 기본 설계
- 모델 작성 및 검증


<br/>

#### ⬇️ 하위 CASE

소프트웨어 개발 후반 단계에서 사용하는 도구이다.

#### 지원 기능

- 상세 설계
- 구현
- 테스트
- 코드 생성


<br/>

### ✨ CASE 주요 기능

- 그래픽 환경을 통한 모델 작성 지원
- 소프트웨어 생명주기 단계 연결
- 개발 모형 지원
- 문서 자동화
- 데이터 및 작업 공유 지원
- 변경 사항 추적 지원

## 📑 HIPO ( Hierarchy Input Process Output )

### 📖 정의

HIPO는 **시스템 분석 및 설계 과정에서 입력(Input), 처리(Process), 출력(Output)의 기능을 계층적으로 표현하는 문서화 기법**이다.

하향식 소프트웨어 개발을 위한 문서화 도구로 사용된다.


#### ✨ 특징

- 시스템 기능을 계층 구조로 표현한다.
- 입력, 처리, 출력 관계를 명확하게 나타낸다.
- 기호와 도표를 사용하여 이해하기 쉽다.
- 기능과 자료의 의존 관계를 표현할 수 있다.
- 변경 및 유지보수가 용이하다.


<br/>

### 📊 HIPO Chart 종류

#### 1. 가시적 도표 ( Visual Table of Contents )

시스템 전체 기능과 흐름을 계층 구조로 표현한다.

#### 2. 총체적 도표 ( Overview Diagram )

프로그램 기능에 대한 전반적인 정보를 제공한다.

- 입력
- 처리
- 출력


#### 3. 세부적 도표 ( Detail Diagram )

총체적 도표의 기능을 구성하는 세부 내용을 상세하게 표현한다.

| 분석 도구 | 핵심 내용 |
|---|---|
| DFD | 자료 흐름과 처리 과정을 표현하는 구조적 분석 도구 |
| 자료사전 | DFD에 표현된 자료를 상세하게 정의하는 도구 |
| CASE | 소프트웨어 개발 과정 자동화를 지원하는 도구 |
| HIPO | 입력·처리·출력 기능을 계층 구조로 표현하는 문서화 도구 |
