# 💻 01. 소프트웨어 아키텍처

## 📖 소프트웨어 아키텍처

### 📖 정의

소프트웨어 아키텍처(Software Architecture)는 **소프트웨어를 구성하는 요소(Component)와 요소 간의 관계, 그리고 외부에 드러나는 특성을 표현한 시스템의 기본 구조**이다.

소프트웨어를 설계하고 개발하기 위한 **지침과 원칙**을 제공한다.

 시스템의 전체 구조와 구성 요소 간의 관계를 정의한다.

#### ✨ 특징

- 시스템의 전체 구조를 정의한다.
- 이해관계자 간의 의사소통 도구로 활용된다.
- 기능 요구사항과 비기능 요구사항을 함께 고려한다.
- 소프트웨어 품질과 유지보수성에 큰 영향을 미친다.
- 개발 초기 단계에서 시스템의 골격을 결정한다.

> 💡 소프트웨어 아키텍처는 **'소프트웨어의 설계도'** 라고 생각하면 이해하기 쉽다.

## 🎯 소프트웨어 아키텍처의 필요성

소프트웨어 아키텍처는 시스템의 품질을 높이고 개발을 효율적으로 수행하기 위해 필요하다.

### ✨ 필요성

- 이해관계자 간의 의사소통을 원활하게 한다.
- 시스템 구조를 체계적으로 설계할 수 있다.
- 품질 속성을 체계적으로 반영할 수 있다.
- 비기능 요구사항을 효과적으로 반영할 수 있다.
- 유지보수성과 재사용성을 향상시킨다.
- 시스템 변경에 유연하게 대응할 수 있다.

## 📋 기능적 요구사항과 비기능적 요구사항

| 구분 | 설명 |
|---|---|
| 기능적 요구사항 (Functional Requirement) | 시스템이 반드시 수행해야 하는 기능에 대한 요구사항 |
| 비기능적 요구사항 (Non-Functional Requirement) | 성능, 보안, 품질, 신뢰성 등 시스템의 품질과 제약사항에 대한 요구사항 |

> 💡 소프트웨어 아키텍처는 **비기능 요구사항을 중심으로 설계하지만 기능 요구사항도 함께 고려한다.**

## 🛠️ 소프트웨어 아키텍처 설계

### 📖 정의

소프트웨어 아키텍처 설계는 **사용자의 요구사항을 만족하는 시스템 구조를 결정하고, 구성 요소와 인터페이스를 설계하는 과정**이다.

애플리케이션의 분할 방법과 각 모듈의 기능, 모듈 간 인터페이스 등을 결정한다.

#### ✨ 특징

- 기능 요구사항과 비기능 요구사항을 함께 고려한다.
- 시스템 구조와 컴포넌트를 정의한다.
- 모듈 간 인터페이스를 설계한다.
- 소프트웨어 품질을 확보하기 위한 기반이 된다.

## 📐 소프트웨어 아키텍처 설계의 기본 원리

> 📝 암기 : **모추단정**

소프트웨어 아키텍처는 다음 네 가지 원리를 기반으로 설계한다.

| 원리 | 설명 |
|---|---|
| 모듈화 (Modularity) | 기능을 모듈 단위로 분리하여 관리한다. |
| 추상화 (Abstraction) | 중요한 개념만 표현하고 세부 사항은 숨긴다. |
| 단계적 분해 (Stepwise Refinement) | 상위 개념부터 하위 개념으로 점차 구체화한다. |
| 정보 은닉 (Information Hiding) | 내부 구현을 숨기고 필요한 정보만 공개한다. |

<br/>

### 📦 모듈화 ( Modularity )

#### 📖 정의

모듈화는 **시스템의 기능을 독립적인 모듈 단위로 분리하는 것**이다.

#### ✨ 특징

- 유지보수가 쉬워진다.
- 재사용성이 향상된다.
- 오류의 영향을 최소화할 수 있다.
- 인터페이스가 단순해진다.
- 공통 모듈을 통해 재사용성을 향상시킨다.

> 💡 모듈 수가 증가하면 모듈 크기는 작아지고, 통합 비용은 증가할 수 있다.

<br/>

### 🎯 추상화 ( Abstraction )

#### 📖 정의

추상화는 **중요한 개념만 표현하고 불필요한 세부 사항은 숨기는 것**이다.

복잡한 시스템을 단순하게 표현하여 설계를 쉽게 한다.

#### ✨ 추상화의 유형

> 📝 암기 : **과데제**

| 유형 | 설명 |
|---|---|
| 과정 추상화 | 전체 처리 흐름만 표현한다. |
| 데이터 추상화 | 데이터의 세부 내용을 숨기고 구조만 표현한다. |
| 제어 추상화 | 제어 흐름이나 이벤트 처리만 표현한다. |

<br/>

### 📉 단계적 분해 ( Stepwise Refinement )

#### 📖 정의

단계적 분해는 **상위 개념부터 시작하여 점차 세부적인 내용으로 분해하는 설계 방법**이다.

Niklaus Wirth가 제안한 **하향식(Top-Down) 설계 기법**이다.

#### ✨ 특징

- 추상화를 반복하여 점진적으로 구체화한다.
- 기능부터 설계한 후 알고리즘과 자료구조를 결정한다.
- 시스템을 계층적으로 설계할 수 있다.

<br/>

### 🔒 정보 은닉 ( Information Hiding )

#### 📖 정의

정보 은닉은 **모듈 내부의 구현 내용을 외부에 숨기고 필요한 인터페이스만 공개하는 기법**이다.

#### ✨ 특징

- 모듈 간의 결합도를 낮춘다.
- 유지보수가 쉬워진다.
- 변경이 다른 모듈에 영향을 주지 않는다.
- 독립적인 개발이 가능하다.

> 💡 다른 모듈은 내부 구현이 아닌 **인터페이스를 통해서만 접근**한다.

## 🏆 소프트웨어 아키텍처의 품질 속성

### 📖 정의

소프트웨어 아키텍처의 품질 속성(Quality Attributes)은 **아키텍처가 이해관계자의 요구사항을 만족하는 수준의 품질을 보장하는지 평가하기 위한 기준**이다.

품질 속성은 **시스템 측면, 비즈니스 측면, 아키텍처 측면**으로 구분한다.

<br/>

### 🖥️ 시스템 측면

시스템이 사용자에게 안정적이고 효율적인 서비스를 제공하기 위해 필요한 품질 요소이다.

암기:가변성 보사시

| 품질 속성 | 설명 |
|---|---|
| 성능 (Performance) | 요청을 빠르고 효율적으로 처리하는 능력 |
| 보안 (Security) | 허가되지 않은 접근을 방지하고 데이터를 보호하는 능력 |
| 가용성 (Availability) | 사용자가 요구한 기능을 정상적으로 제공하는 능력 |
| 사용성 (Usability) | 사용자가 쉽고 편리하게 사용할 수 있는 정도 |
| 변경 용이성 (Modifiability) | 요구사항 변경이나 기능 수정이 쉬운 정도 |
| 확장성 (Scalability) | 시스템의 용량이나 처리 능력을 쉽게 확장할 수 있는 능력 |

> 💡 그 밖에도 **시험성(Testability), 안정성(Stability), 배치성(Deployability)** 등이 있다.

<br/>

### 💼 비즈니스 측면

비즈니스 목표와 경제성을 고려한 품질 요소이다.

| 품질 속성 | 설명 |
|---|---|
| 시장 적시성 (Time to Market) | 정해진 일정에 맞추어 시스템을 출시하는 것 |
| 비용과 혜택 (Cost & Benefit) | 개발 비용과 유지보수 비용 등을 고려하는 것 |
| 예상 시스템 수명 (Lifetime) | 시스템을 얼마나 오랫동안 사용할 것인지 고려하는 것 |

> 💡 시스템의 수명이 길수록 **변경 용이성**과 **확장성**을 더욱 중요하게 고려한다.

<br/>

### 🏗️ 아키텍처 측면

아키텍처 자체가 갖추어야 하는 품질 요소이다.

| 품질 속성 | 설명 |
|---|---|
| 개념적 무결성 (Conceptual Integrity) | 시스템 전체의 일관성을 유지하는 것 |
| 정확성 (Correctness) | 요구사항을 정확하게 만족하는 것 |
| 완결성 (Completeness) | 필요한 기능과 제약사항을 모두 반영하는 것 |
| 구축 가능성 (Buildability) | 모듈을 효율적으로 개발하고 통합할 수 있는 정도 |

> 💡 그 밖에도 **변경성(Modifiability), 시험성(Testability), 적응성(Adaptability), 대체성(Substitutability)** 등이 있다.

## ⚙️ 소프트웨어 아키텍처 설계 과정

### 📖 정의

소프트웨어 아키텍처 설계 과정은 **요구사항을 분석하여 시스템 구조를 설계하고 검증하는 과정**이다.

> 📝 암기 : **목타패구검**

소프트웨어 아키텍처 설계는 다음 **5단계**로 진행된다.

| 단계 | 설명 |
|---|---|
| 설계 목표 설정 | 요구사항과 비즈니스 목표를 분석하여 설계 목표를 설정 |
| 시스템 타입 결정 | 시스템 특성에 맞는 구조와 아키텍처 패턴을 결정 |
| 아키텍처 패턴 적용 | 적절한 아키텍처 패턴을 선택하여 시스템 구조를 설계 |
| 서브시스템 구체화 | 서브시스템의 기능과 인터페이스를 정의 |
| 검토 | 요구사항 반영 여부와 설계 원칙 준수 여부를 검토 |

<br/>

### 1️⃣ 설계 목표 설정

시스템의 개발 방향을 명확하게 하기 위해 **요구사항과 비즈니스 목표를 분석**하고 설계 목표를 설정한다.

<br/>

### 2️⃣ 시스템 타입 결정

시스템의 특성을 분석하여 적절한 **시스템 구조와 아키텍처 패턴**을 선택한다.

<br/>

### 3️⃣ 아키텍처 패턴 적용

선택한 아키텍처 패턴을 적용하여 **표준 아키텍처를 설계**한다.

<br/>

### 4️⃣ 서브시스템 구체화

각 서브시스템의 **기능과 인터페이스**, 그리고 서브시스템 간의 상호작용을 정의한다.

<br/>

### 5️⃣ 검토

설계 결과가 **요구사항을 충족하는지**, 설계 목표와 요구사항을 만족하는지 검토한다.

## 🖥️ 시스템 타입

### 📖 정의

시스템 타입(System Type)은 **시스템의 동작 방식과 처리 특성에 따라 시스템을 분류한 것**이다.

소프트웨어 아키텍처를 설계할 때 시스템의 특성에 맞는 구조와 아키텍처 패턴을 선택하기 위한 기준으로 활용된다.

#### ✨ 시스템 타입의 종류

| 시스템 타입 | 설명 |
|---|---|
| 대화형 시스템 (Interactive System) | 사용자의 요청에 따라 시스템이 즉시 반응하는 시스템 |
| 이벤트 중심 시스템 (Event-Driven System) | 외부에서 발생한 이벤트를 중심으로 동작하는 시스템 |
| 변환형 시스템 (Transformational System) | 입력 데이터를 처리하여 결과를 출력하는 시스템 |
| 객체 영속형 시스템 (Object-Persistent System) | 데이터베이스 등을 이용하여 객체를 저장·검색·갱신하는 시스템 |

> 💡 시스템 타입을 결정한 후, 그 특성에 맞는 **아키텍처 패턴**을 선택한다.

## 📜 협약에 의한 설계 ( Design by Contract )

### 📖 정의

협약에 의한 설계(Design by Contract)는 **소프트웨어 컴포넌트 간의 정확한 인터페이스를 명세하기 위해 계약(Contract)의 형태로 조건을 정의하는 설계 기법**이다.

컴포넌트나 클래스가 반드시 지켜야 하는 조건을 명확하게 정의하여 신뢰성을 높인다.

#### ✨ 특징

- 컴포넌트의 인터페이스를 명확하게 정의한다.
- 오류를 사전에 방지할 수 있다.
- 클래스 간의 책임을 명확하게 한다.
- 소프트웨어의 신뢰성을 향상시킨다.

<br/>

### 📋 계약의 구성 요소

| 구성 요소 | 설명 |
|---|---|
| 선행 조건 (Precondition) | 연산이 수행되기 전에 반드시 만족해야 하는 조건 |
| 결과 조건 (Postcondition) | 연산이 수행된 후 반드시 만족해야 하는 조건 |
| 불변 조건 (Invariant) | 객체가 존재하는 동안 항상 유지되어야 하는 조건 |

## 🏗️ 소프트웨어 아키텍처 4+1 View

### 📖 정의

소프트웨어 아키텍처 **4+1 View 모델**은 **시스템을 서로 다른 관점(View)에서 표현하여 아키텍처를 이해하기 쉽게 만드는 설계 방법**이다.

Philippe Kruchten이 제안한 모델로, **4개의 구조적 관점과 이를 검증하는 1개의 유스케이스 관점**으로 구성된다.

> 💡 **'+1'은 유스케이스 뷰를 의미한다.**

<br/>

### 📊 4+1 View 구성

암기:유논프구배

| 뷰(View) | 설명 | 주요 이해관계자 |
|---|---|---|
| 유스케이스 뷰 (Use Case View) | 시스템의 기능 요구사항과 시나리오를 표현하며 다른 뷰를 검증 | 사용자, 설계자, 개발자, 테스트 담당자 |
| 논리 뷰 (Logical View) | 시스템의 기능과 객체, 클래스 구조를 표현 | 설계자, 개발자 |
| 프로세스 뷰 (Process View) | 동시성, 병행 처리, 성능 등의 동적인 구조를 표현 | 개발자, 시스템 통합자 |
| 구현 뷰 (Implementation View) | 컴포넌트와 모듈의 구성 및 의존 관계를 표현 | 개발자 |
| 배포 뷰 (Deployment View) | 소프트웨어가 하드웨어에 어떻게 배치되는지를 표현 | 개발자, 시스템 엔지니어 |

## 💰 소프트웨어 아키텍처 비용 평가 모델

### 📖 정의

소프트웨어 아키텍처 비용 평가 모델은 **아키텍처가 품질 속성을 얼마나 만족하는지 평가하고, 가장 적합한 아키텍처를 선택하기 위한 평가 기법**이다.

> 📝 암기 : **사카린 (SAAM → ATAM → CBAM → ADR → ARID)**

<br/>

### 📊 비용 평가 모델의 종류

| 모델 | 설명 |
|---|---|
| SAAM (Software Architecture Analysis Method) | 변경 용이성과 기능성을 중심으로 평가하는 기법 |
| ATAM (Architecture Tradeoff Analysis Method) | 품질 속성과 품질 속성 간의 상충 관계(Trade-off)를 평가하는 기법 |
| CBAM (Cost Benefit Analysis Method) | ATAM을 기반으로 비용과 효과를 함께 고려하는 경제성 평가 기법 |
| ADR (Active Design Review) | 아키텍처 구성 요소 간의 응집도를 평가하는 기법 |
| ARID (Active Reviews for Intermediate Designs) | 전체 아키텍처보다 특정 부분의 설계를 집중적으로 평가하는 기법 |

> 💡 **CBAM은 ATAM을 기반으로 비용 대비 효과(Benefit)를 함께 분석하는 것이 핵심이다.**

<br/>

### 📌 아키텍처 품질 속성

아키텍처 비용 평가에서는 **품질 속성(Quality Attribute)** 이 중요한 평가 기준이 된다.

품질 속성은 시스템이 만족해야 하는 품질 요구사항을 의미하며, 최적의 아키텍처를 선택하기 위한 핵심 요소이다.

### 📌 시스템 설계 및 평가

| 구분 | 핵심 내용 |
|---|---|
| 시스템 타입 | 대화형, 이벤트 중심, 변환형, 객체 영속형 |

| ATAM | 품질 속성 및 Trade-off 평가 |

| ADR | 구성 요소 간의 아키텍처 구성 요소의 적합성을 평가 |
